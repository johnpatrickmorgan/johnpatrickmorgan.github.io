URL Pattern Matching

May 11, 2016 in Swift

When implementing deeplinks for an iOS app recently, I wondered if Swift’s pattern matching could be used to match URLs rather than using a URL routing library. This is an account of the different approaches I tried, and the one I finally settled on.

The majority of URL routing libraries use a pattern syntax that enables you to match URL path elements in one of three ways:
 

  • Equality: Path element must match the pattern expression
  • Wildcard: Path element can be anything at all, and will be discarded
  • Value-binding: Path element can be anything at all, and will be made available in a parameters dictionary

  • The parameter name userId is repeated and is susceptible to typos.
  • The pattern matching is abstracted away using special syntax.

As it happens Swift’s built-in pattern matching can be used for each of the three pattern types. Here’s an example of all three:

In fact, the expression pattern is more powerful than a simple equality test, as we can define our own matching logic using the pattern matching operator. Swift’s pattern-matching and switch statement would seem a natural fit for matching a URL to a pattern, so I decided to investigate a Swift pattern-matching-based approach to URL routing.

Let’s assume the path part of a URL has been parsed into an Array called pathElements, e.g., https://myhost.com/user/14253/profile would give ["user", "14253", "profile"]. There is no pattern matching for Arrays built in to Swift, so it needs to be added somehow…

Approach 3: Linked List

Approach 4: Counted

My final approach was a compromise between approaches 2 and 3. What was needed was an enum that could represent arbitrary numbers of elements without needing too many layers of nested enums. Enter Counted:

Counted is an enum where each case has a different number of elements as associated values. It can be initialized with an Array, and just like List, there’s an indirect case that enables large arrays to be represented via nested Counted enums. Unlike List, a layer of nesting is only required for every 10 elements, which makes things easier to read:

The names of enum cases cannot begin with a digit and must be unique, so each case name comprises the count prefixed with N: unfortunately, they can’t be further abbreviated. Counted enables us to pattern match paths with any number of elements, and supports expression, wildcard and value-binding patterns for its associated values.
